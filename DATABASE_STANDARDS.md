# Database Standard

Version: 1.1.0  
Status: Final  
Owner: Project maintainers  
Applies to: All projects based on `project-master-template`

---

## 1. Purpose

This document defines the default database standards for projects using this template.

The goal is to ensure that every project has:

- Clear referential integrity
- Sensible normalization
- Explicit relationships
- Safe handling of user and personal data
- Predictable naming conventions
- Reviewable database changes
- Good performance through intentional indexing
- A schema that can be understood by humans, not just generated by tools

This standard is especially important when AI coding assistants are used. AI-generated database schemas must follow this document unless a deviation is explicitly documented.

---

## 2. Core Principles

### 2.1 The database is part of the application contract

The database schema is not just storage. It defines business rules, data ownership, security boundaries and long-term maintainability.

### 2.2 Prefer explicit constraints over implicit assumptions

Rules that must always be true should be enforced by the database where practical.

Examples:

- Required fields should be `NOT NULL`
- Unique values should use `UNIQUE` constraints
- Relationships should use foreign keys
- Valid states should use controlled values or check constraints where appropriate

### 2.3 Use relational modelling by default

Projects should use proper relational modelling unless there is a clear reason not to.

Use:

- Primary keys
- Foreign keys
- Join tables
- Unique constraints
- Indexes
- Normalized entities

Avoid storing relational data as unstructured JSON unless the data is genuinely flexible, external, experimental or schemaless.

### 2.4 Optimize for correctness first, convenience second

A slightly more structured schema is better than a convenient schema that creates duplicate, inconsistent or ambiguous data.

## 3. Supported Database Types

This standard is written primarily for relational databases such as:

- PostgreSQL
- SQLite
- MySQL/MariaDB
- SQL Server
- Oracle

Project-specific notes may be added for specific platforms such as Neon Postgres, SQLite, SQLCipher, Supabase or Cloudflare D1.

## 4. Naming Conventions

### 4.1 General naming

Use `snake_case` for database identifiers.

Examples:

```sql
users
user_profiles
created_at
project_id
```

Avoid:

```sql
UserProfile
createdAt
ProjectID
```

### 4.2 Table names

Default: use singular table names.

Examples:

```sql
user
project
task
meal_item
```

### 4.3 Primary keys

Default primary key column:

```sql
id
```

Default primary key type:

```sql
uuid
```

UUID is the global default because it works well for PostgreSQL, SQLite, local-first apps, synchronization, imports and distributed systems.

Integer identity keys may only be used when there is a specific documented reason.

### 4.4 Foreign keys

Foreign keys should use the referenced table name in singular form plus `_id`.

Examples:

```sql
user_id
project_id
meal_id
food_id
```

### 4.5 Join tables

Join tables should normally use both entity names.

Examples:

```sql
user_projects
project_members
meal_foods
person_tags
```

A join table should have:

- Foreign keys to both parent tables
- A composite unique constraint
- Indexes on foreign key columns
- Optional metadata columns if the relationship has meaning

Example:

```sql
create table project_members (
  project_id uuid not null references projects(id),
  user_id uuid not null references user(id),
  role text not null,
  created_at timestamptz not null default now(),
  primary key (project_id, user_id)
);
```

## 5. Schema Design and Normalization

### 5.1 Minimum normalization level

Schemas should normally follow at least Second Normal Form (2NF), and preferably Third Normal Form (3NF).

This means:

- Avoid repeating groups of columns
- Avoid duplicated facts across multiple tables
- Avoid storing derived values as source-of-truth unless justified
- Keep facts about different entities in separate tables
- Use relationship tables for many-to-many relationships

### 5.2 When denormalization is allowed

Denormalization is allowed only when there is a clear reason.

Valid reasons may include:

- Performance optimization based on actual query patterns
- Snapshotting historical values
- Audit/event history
- External API payload preservation
- Reporting/read models
- Offline/local-first synchronization

Any denormalized column or table should be documented.

Example:

```md
Denormalized field: `meal_items.calculated_kcal`
Reason: preserves historical nutrition calculation even if source food data changes later.
```

### 5.3 Avoid unclear JSON storage

JSON columns may be used for flexible or external data, but not as a shortcut to avoid modelling important entities.

Good use of JSON:

- Raw external API response
- UI preferences
- Experimental metadata
- Provider-specific payloads

Bad use of JSON:

- Storing users, tasks, meals, permissions or relationships as blobs
- Avoiding foreign keys
- Hiding queryable business data

## 6. Keys, Relationships and Referential Integrity

### 6.1 Every table must have a primary key

Every table must have a primary key unless there is a documented technical reason.

Join tables may use a composite primary key.

### 6.2 Foreign keys are required by default

If one row refers to another entity, the database should enforce that relationship with a foreign key.

Example:

```sql
user_id uuid not null references user(id)
```

### 6.3 Define delete behavior explicitly

Every foreign key relationship should have a deliberate delete strategy.

Common options:

- `RESTRICT`: prevent deletion while child records exist
- `CASCADE`: delete child records automatically
- `SET NULL`: preserve child records but remove relationship
- Soft delete: mark records as deleted instead of removing them

Do not use cascade delete casually. It can be dangerous for user data.

Recommended default:

- Use `RESTRICT` for important business data
- Use `CASCADE` for purely dependent data
- Use soft delete for user-owned records where recovery or audit matters

### 6.4 Avoid orphaned records

The schema should not allow orphaned records unless explicitly documented.

---

## 7. Constraints

Use constraints to protect data quality.

Recommended constraints:

- `NOT NULL` for required fields
- `UNIQUE` for natural uniqueness
- Foreign keys for relationships
- Check constraints for limited valid values
- Composite unique constraints for scoped uniqueness

Examples:

```sql
unique(user_id, slug)
unique(project_id, name)
check (amount >= 0)
check (status in ('draft', 'active', 'archived'))
```

Application validation is not a replacement for database constraints.

---

## 8. Indexing Standard

### 8.1 Index intentionally

Indexes should support known access patterns.

Common reasons to add an index:

- Foreign key lookup
- Filtering
- Sorting
- Search
- Unique lookup
- Frequent joins

### 8.2 Index foreign keys

Foreign key columns should usually be indexed.

Example:

```sql
create index idx_meals_user_id on meals(user_id);
```

### 8.3 Use composite indexes for common query patterns

Example:

```sql
create index idx_meal_user_date on meal(user_id, meal_date);
```

Use composite indexes when queries commonly filter by the same combination of columns.

### 8.4 Avoid index hoarding

Do not add indexes "just in case".

Indexes improve reads but can slow writes and increase storage.

Each non-obvious index should have a reason.

Example:

```md
Index: `idx_meals_user_date`
Reason: daily meal overview filters by user and date on every dashboard load.
```

## 9. Audit Fields and Timestamps

### 9.1 Standard fields

Most business tables should include:

```sql
created_at
updated_at
```

Optional fields depending on project needs:

```sql
deleted_at
created_by
updated_by
deleted_by
```

### 9.2 Timestamp rules

Use timezone-aware timestamps where supported.

Preferred for PostgreSQL:

```sql
timestamptz
```

Use UTC internally unless a project has a documented reason to do otherwise.

### 9.3 Soft delete

Use `deleted_at` for user-owned data by default.

Use `deleted_at` when records should be hidden but retained.

Soft-deleted records should usually be excluded from normal queries.

If soft delete is used, define whether related records are also soft-deleted or retained.

## 10. Security and Privacy

### 10.1 Data minimization

Do not store data that the application does not need.

Before adding personal or sensitive fields, ask:

- Why do we need this?
- How long do we need to keep it?
- Who can access it?
- Can it be derived instead of stored?
- Can it be anonymized or reduced?

### 10.2 Sensitive data

Sensitive data must be identified during schema design.

Examples:

- Health data
- Personal notes
- Private relationship data
- Authentication identifiers
- Email content
- Calendar content
- Location data
- Payment or billing data

### 10.3 Secrets must never be stored in ordinary tables

Do not store API keys, access tokens, refresh tokens or secrets in plain text.

Use:

- Environment variables
- Secret managers
- Encrypted storage
- Provider-specific secure token storage

### 10.4 No sensitive data in logs

Database errors, debug logs and audit logs must not expose sensitive user data.

### 10.5 Authorization must be considered in schema design

For multi-user or workspace-based systems, the schema must make ownership clear.

Common ownership columns:

```sql
user_id
workspace_id
organization_id
```

Every user-owned table should have a clear access boundary.

### 10.6 Row-level security

For PostgreSQL projects that expose database access through client-facing APIs, consider Row Level Security.

If RLS is used, policies must be documented and tested.

## 11. Default Database and Tooling

### 11.1 Default database choices

Default choices:

- Web applications: PostgreSQL
- Local-first mobile applications: SQLite / SQLCipher
- Sensitive local-first mobile applications: SQLCipher strongly preferred

Other databases may be used, but the reason must be documented.

### 11.2 Default TypeScript database tool

Default ORM / schema tool for TypeScript projects:

```text
Drizzle
```

Prisma may be used only when a project has a specific reason. The reason should be documented.

### 11.3 Project-specific database notes

Each project should document:

- Database engine
- Hosting/provider
- Migration tool
- Local development setup
- Production setup
- Backup/restore expectations

## 12. Migrations

### 12.1 All schema changes must use migrations

Database changes must be made through migration files.

Do not apply undocumented manual production changes.

### 12.2 Migrations should be reviewable

Migration files should be committed to version control.

They should be small enough to review.

### 12.3 Destructive migrations require special care

Destructive changes include:

- Dropping tables
- Dropping columns
- Changing column types
- Deleting data
- Backfilling large datasets

Destructive migrations must include:

- Reason
- Risk
- Rollback plan where practical
- Data migration plan if needed

### 12.4 Migration history

Migration history should be preserved.

Do not rewrite applied migration history unless the project is still disposable and not deployed.

---

## 13. Seed Data and Test Data

### 13.1 Seed data

Seed data should be deterministic and safe.

Use seed data for:

- Local development
- Demo mode
- Automated tests
- Required reference data

### 13.2 No real personal data in tests

Test data must not contain real personal information unless explicitly approved and protected.

Use realistic fake data instead.

### 13.3 Test database constraints

Automated tests should verify important database constraints where practical.

Examples:

- Cannot create child record without parent
- Cannot insert duplicate scoped slug
- Cannot insert invalid status
- Deleting parent behaves as expected

---

## 14. External Data and API Imports

When importing data from external APIs, separate source data from internal normalized data where appropriate.

Recommended pattern:

- Store normalized application entities
- Optionally store raw external payloads separately for debugging/audit
- Track source/provider identifiers

Example columns:

```sql
source_provider
source_external_id
source_updated_at
```

Use unique constraints to prevent duplicate imports.

Example:

```sql
unique(source_provider, source_external_id)
```

## 15. History and Change Tracking

### 15.1 Project history

Projects that need historical tracking should explicitly decide between:

- Simple audit fields
- Soft delete
- Audit log table
- Event sourcing
- Snapshot tables
- Versioned records

### 15.2 Audit log table

Every project must include an audit log strategy.

For small/local projects, a simple audit table is enough. For sensitive, shared or production systems, audit logging should be treated as mandatory application infrastructure.

Use an audit log to know who changed what and when.

Example structure:

```sql
audit_log (
  id,
  entity_type,
  entity_id,
  action,
  changed_by,
  changed_at,
  before_json,
  after_json
)
```

Do not store sensitive fields in audit logs unless there is a clear reason and appropriate protection.

### 15.3 Historical data rule

Historical facts must not be silently rewritten.

If a value was true at the time an event happened, preserve that value when future reference data changes.

Examples:

- A logged meal should preserve calculated calories even if the source food entry is updated later.
- An invoice should preserve the original price even if the product price changes later.
- A meeting summary should preserve the version that was sent or approved.

Use snapshot fields, versioned records or history tables when historical meaning matters.

### 15.4 Snapshotting

Use snapshots when historical meaning must be preserved.

Example:

A meal item may store calculated calories at the time it was logged, even if the source food entry changes later.

## 16. Documentation Requirements

Every project should document:

- Main entities
- Important relationships
- Ownership model
- Sensitive tables/columns
- External data sources
- Deletion strategy
- Migration tool
- Known denormalized fields
- Indexes that are not obvious

A simple ERD or relationship sketch is recommended for all non-trivial projects.

## 17. AI Agent Rules

AI coding assistants must follow these rules when creating or modifying database schemas.

### 17.1 Do not create schema casually

Before creating tables, the assistant should identify:

- Entities
- Relationships
- Ownership
- Required fields
- Constraints
- Indexes
- Privacy concerns
- Migration impact

### 17.2 Every generated schema must include

- Primary keys
- Foreign keys where applicable
- Required `NOT NULL` constraints
- Unique constraints where applicable
- Indexes for common queries
- Delete behavior
- Timestamps where appropriate

### 17.3 Do not fake relational integrity in application code

If the database supports foreign keys, use them.

### 17.4 Ask before storing sensitive data

The assistant must ask or document why sensitive data is stored.

### 17.5 No undocumented shortcuts

The assistant must not use JSON blobs, duplicated columns, missing constraints or missing relationships just to move faster.

### 17.6 Explain deviations

Any deviation from this standard must be documented in the pull request, issue or schema notes.

## 18. Review Checklist

Use this checklist before approving database changes.

### Structure

- [ ] Every table has a primary key
- [ ] Foreign keys are used for relationships
- [ ] Many-to-many relationships use join tables
- [ ] Required fields are `NOT NULL`
- [ ] Unique business rules are enforced with constraints
- [ ] Delete behavior is explicit

### Normalization

- [ ] Schema is at least 2NF
- [ ] Schema is preferably 3NF
- [ ] No unnecessary duplicated facts
- [ ] Derived values are justified
- [ ] JSON fields are justified

### Indexes

- [ ] Foreign keys are indexed where useful
- [ ] Common filters and joins are indexed
- [ ] Composite indexes match real query patterns
- [ ] No unnecessary indexes

### Security and Privacy

- [ ] Sensitive data is identified
- [ ] Ownership/access boundary is clear
- [ ] No secrets are stored in plain text
- [ ] Logs will not expose sensitive data
- [ ] Retention/deletion strategy is considered

### Migrations

- [ ] Migration file exists
- [ ] Migration is reviewable
- [ ] Destructive changes are documented
- [ ] Rollback/data migration risk is considered

### Documentation

- [ ] Main entities are documented
- [ ] Relationships are documented
- [ ] Denormalization is explained
- [ ] External data sources are documented

## 19. Example: Good Minimal Schema

Example for a simple calorie tracking feature.

```sql
create table food (
  id uuid primary key,
  name text not null,
  source_provider text,
  source_external_id text,
  kcal_per_100g numeric(8,2) not null,
  protein_per_100g numeric(8,2),
  carbs_per_100g numeric(8,2),
  fat_per_100g numeric(8,2),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  unique(source_provider, source_external_id)
);

create table meal (
  id uuid primary key,
  user_id uuid not null references user(id) on delete cascade,
  meal_date date not null,
  meal_type text not null,
  name text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table meal_item (
  id uuid primary key,
  meal_id uuid not null references meal(id) on delete cascade,
  food_id uuid not null references food(id),
  grams numeric(8,2) not null check (grams > 0),
  calculated_kcal numeric(8,2) not null,
  created_at timestamptz not null default now()
);

create index idx_meal_user_date on meal(user_id, meal_date);
create index idx_meal_item_meal_id on meal_item(meal_id);
create index idx_meal_item_food_id on meal_item(food_id);

create table audit_log (
  id uuid primary key,
  entity_type text not null,
  entity_id uuid not null,
  action text not null,
  changed_by uuid,
  changed_at timestamptz not null default now(),
  before_json jsonb,
  after_json jsonb
);

create index idx_audit_log_entity on audit_log(entity_type, entity_id);
create index idx_audit_log_changed_at on audit_log(changed_at);
```

Notes:

- `food` stores nutrition per 100g
- `meal` belongs to a user
- `meal_item` connects food to meals
- `calculated_kcal` is intentionally snapshotted
- Indexes support daily meal overview and joins

## 20. Final Decisions

These are the default decisions for this template.

| Topic | Decision |
|---|---|
| Table names | Singular, `snake_case` |
| Primary keys | `id uuid primary key` |
| Default TypeScript ORM/schema tool | Drizzle |
| Web database | PostgreSQL |
| Local-first mobile database | SQLite / SQLCipher |
| Sensitive local-first mobile database | SQLCipher strongly preferred |
| Soft delete | Default for user-owned data |
| Foreign keys | Required by default |
| Normalization | At least 2NF, preferably 3NF |
| ERD / relationship sketch | Required before non-trivial database work |
| Audit log | Required for all projects |
| Historical facts | Must be preserved through snapshots/versioning/history where relevant |
| JSON columns | Allowed only when justified |
| Manual production schema changes | Not allowed |

## 21. ERD Requirement

Before non-trivial database work starts, the project must include a simple entity relationship sketch.

This can be a lightweight text diagram.

Example:

```text
user
 └── meal
      └── meal_item
           └── food
```

For larger projects, use Mermaid, Excalidraw or another diagram format.

Minimum ERD content:

- Main entities
- Ownership boundaries
- One-to-many relationships
- Many-to-many relationships
- External data sources
- Sensitive entities
